# Research: シンプル将棋アプリ

**Branch**: `001-simple-shogi-app` | **Date**: 2026-05-22

## Decision Log

---

### Decision 1: フロントエンドフレームワーク

**Decision**: React 18 + Vite 5 + TypeScript 5  
**Rationale**: ユーザー指定。Viteは開発サーバー起動が高速（HMR）でビルドも最適化される。ReactはコンポーネントベースでUIを宣言的に記述でき、将棋盤のような複雑なUIに適している。TypeScriptは駒の型・座標・ゲーム状態などの厳密な型定義が可能で、将棋ルールのバグを防ぎやすい。  
**Alternatives considered**: Vue 3（同様に適切だが指定なし）、vanilla TS（依存が少ないが状態管理が複雑になる）

---

### Decision 2: 状態管理

**Decision**: `useReducer` + Context（外部ライブラリなし）  
**Rationale**: 将棋のゲーム状態は明確に定義されたアクション（MOVE_PIECE, DROP_PIECE, RESIGN, RESET等）によって遷移する。useReducerはReducer関数として純粋に記述でき、テストが容易。Reduxは過剰エンジニアリング。ZustandはuseReducerで十分な規模に対して不要な依存を増やす。  
**Alternatives considered**: Redux Toolkit（大規模アプリ向け、過剰）、Zustand（シンプルだが外部依存追加）、useState複数（状態の一貫性維持が困難）

---

### Decision 3: ゲームエンジンの分離

**Decision**: `src/game/` ディレクトリに純粋TypeScript関数として実装（Reactへの依存ゼロ）  
**Rationale**: 将棋ルールエンジンをReactコンポーネントから完全分離することで、VitestによるユニットテストをDOMなしで実行できる。合法手計算・詰み検出・千日手判定などの複雑なロジックは独立してテストすべき。副作用なしの純粋関数として実装することで予測可能性とデバッグ性が向上する。  
**Alternatives considered**: クラスベース実装（テストが複雑になる）、コンポーネント内に直書き（テスト不可・責務混在）

---

### Decision 4: テストフレームワーク

**Decision**: Vitest（ゲームロジック）+ React Testing Library（コンポーネント）  
**Rationale**: VitestはViteと同じ設定を共有でき、設定コストが最小。Jest互換のAPIなので学習コストが低い。React Testing Libraryはユーザー視点でのコンポーネントテストに最適。  
**Alternatives considered**: Jest（Viteとの統合設定が複雑）、Playwright（E2Eはスコープ外）

---

### Decision 5: 将棋駒の表示方式

**Decision**: Unicode／漢字テキストで駒を表現（SVG画像なし）  
**Rationale**: シンプルな将棋アプリとして、外部アセットなしでHTMLとCSSのみで駒を表現する。各駒に対応する漢字（歩・香・桂・銀・金・角・飛・王/玉・と・成香・成桂・成銀・馬・龍）を使用。後手の駒は180度回転して表示する（CSSで実現）。  
**Alternatives considered**: SVG画像（アセット管理が必要、シンプルでない）、Unicode将棋記号（一部環境で表示不安定）

---

### Decision 6: 千日手の局面ハッシュ

**Decision**: 盤面状態（駒配置＋持ち駒＋手番）を文字列シリアライズしてMapに頻度管理  
**Rationale**: 将棋の千日手判定には「同一局面」の定義が必要（盤面配置 + 持ち駒 + 手番の三要素が全て同一）。文字列シリアライズはシンプルで実装が容易。性能的にも100手程度の対局では問題なし。  
**Alternatives considered**: Zobristハッシュ（高速だが実装複雑、衝突リスクあり）、deepEqual比較（遅い）

---

### Decision 7: 成り選択ダイアログ

**Decision**: モーダルオーバーレイとして実装、ゲーム状態に`pendingPromotion`フィールドを追加  
**Rationale**: 成り・不成りの選択は移動確定前に必要。ゲーム状態に`pendingPromotion: { from, to, piece } | null`を持たせ、選択中は他の操作を無効化する。シンプルなReactモーダルで実現可能。  
**Alternatives considered**: インライン選択UI（盤面レイアウトが複雑になる）、自動成り（ユーザーの選択権を奪う）

---

## 将棋ルール実装チェックリスト

実装時に必ず考慮するルール一覧（research時に収集）:

### 合法手判定（移動）
- [ ] 各駒の基本移動範囲（8種類＋成駒6種類）
- [ ] 盤外への移動禁止
- [ ] 自分の駒があるマスへの移動禁止
- [ ] 飛車・角行・香車の利き（間に駒があれば遮断）
- [ ] 王手放置の禁止（移動後に自玉が王手になる手は非合法）

### 合法手判定（打ち）
- [ ] 駒を持っていない駒種は打てない
- [ ] 駒が既にあるマスには打てない
- [ ] 二歩（同じ筋に歩が2枚）禁止
- [ ] 行き所のない駒（歩・香は1段目、桂は1・2段目に打てない）
- [ ] 打ち歩詰め禁止

### 成り
- [ ] 敵陣（相手側3段）に入る・出る・通過する移動で成り可能
- [ ] 成りを選択できる駒：歩・香・桂・銀・角・飛（金・王は成れない）
- [ ] 行き所のない場所に移動する場合は強制的に成り（桂の1・2段目、歩・香の1段目）
- [ ] 取った成駒は持ち駒として元の駒種に戻る

### 終局条件
- [ ] 詰み：王手がかかっており、全ての合法手が王手回避できない
- [ ] 投了：手番プレイヤーが投了宣言
- [ ] 千日手：同一局面（盤面＋持ち駒＋手番）が4回繰り返し→自動引き分け
- [ ] 持将棋・入玉：対象外（Assumptions参照）
