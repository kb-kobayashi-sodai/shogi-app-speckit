# Feature Specification: 持ち駒エリアの右配置・縦並び・スクロール対応

**Feature Branch**: `006-captured-pieces-right-layout`  
**Created**: 2026-05-22  
**Status**: Draft  
**Input**: User description: "持ち駒を将棋盤の右側に配置して、駒も将棋盤を縦に並べるようにして。長くなるようならスクロールつけて"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 盤面右側で持ち駒を確認する (Priority: P1)

対局中のプレイヤーが、将棋盤の右側に表示された持ち駒エリアを見て、盤面と持ち駒を同時に把握できる。

**Why this priority**: 持ち駒の配置位置は盤面との視覚的な関係性に直結し、ゲームの基本的な使い勝手を決める最重要要素。

**Independent Test**: 将棋アプリを開き、対局画面で盤面の右側に先手・後手の持ち駒エリアが表示されていることを確認するだけで検証可能。

**Acceptance Scenarios**:

1. **Given** 対局画面を開いた状態で、**When** 画面全体を確認すると、**Then** 先手（ユーザー）の持ち駒エリアが将棋盤の右側に、後手の持ち駒エリアが左側に配置されている
2. **Given** 先手・後手それぞれが持ち駒を持っている状態で、**When** 対局画面を表示すると、**Then** 盤面を挟んで左右に各自の持ち駒が配置されている
3. **Given** 持ち駒が0枚の状態で、**When** 持ち駒エリアを確認すると、**Then** エリア自体は表示されており「なし」または空の状態が分かる

---

### User Story 2 - 縦に並んだ持ち駒を識別する (Priority: P2)

プレイヤーが持ち駒エリアを見たとき、駒の種類と枚数が縦に整列されて表示されており、視認しやすい。

**Why this priority**: 横並びより縦並びの方が駒の種類ごとに視線が止まりやすく、素早い判断に繋がる。盤面が縦方向に伸びる将棋の性質にも合致する。

**Independent Test**: 複数種類の持ち駒がある状態で対局画面を表示し、持ち駒エリア内の駒が縦方向に並んで配置されていることを目視確認できる。

**Acceptance Scenarios**:

1. **Given** 複数種類の持ち駒（歩・飛・角など）を持っている状態で、**When** 持ち駒エリアを確認すると、**Then** 駒が縦方向に並んで表示されている
2. **Given** 同じ種類の駒を複数枚持っている状態で、**When** 持ち駒エリアを確認すると、**Then** 枚数が分かる形で縦に整列されている

---

### User Story 3 - 持ち駒が多い場合もスクロールして全て確認する (Priority: P3)

持ち駒が多くなりエリアが縦に長くなった場合、スクロール操作によって画面からはみ出した持ち駒も確認できる。

**Why this priority**: 大量の持ち駒はゲーム終盤にのみ発生するエッジケースだが、その時点でUI が崩れると対局体験を大きく損なう。

**Independent Test**: 多数の持ち駒（7種類以上）がある状態を再現し、持ち駒エリアがスクロール可能になっていること、および全ての駒種が確認できることを検証できる。

**Acceptance Scenarios**:

1. **Given** 持ち駒エリアの表示可能領域を超える量の持ち駒がある状態で、**When** エリア内でスクロール操作を行うと、**Then** 隠れていた持ち駒も表示される
2. **Given** 持ち駒が少なく表示領域に収まる状態で、**When** 持ち駒エリアを確認すると、**Then** スクロールバーは表示されない（または不要な余白が生じない）
3. **Given** スクロール可能な持ち駒エリアがある状態で、**When** スクロールしても、**Then** 将棋盤自体のレイアウトは崩れない

---

### Edge Cases

- 持ち駒が0枚の場合、エリアは空欄または「なし」で表示されレイアウトが崩れないか？
- 先手・後手の両者が大量の持ち駒を持った場合、右側エリアが正しくスクロール対応するか？
- 画面の縦幅が小さい端末（低解像度）でも将棋盤と持ち駒エリアが適切に共存するか？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 先手（ユーザー）の持ち駒エリアは将棋盤の右側に配置されなければならない
- **FR-001b**: 後手の持ち駒エリアは将棋盤の左側に配置されなければならない
- **FR-002**: 持ち駒エリア内の駒は縦方向に並べて表示されなければならない。1行 = 1駒種（駒アイコン + 枚数、例：歩 ×4）とする
- **FR-003**: 持ち駒エリアの高さが将棋盤の高さの半分を超えた場合、エリア内でスクロールできなければならない（上限 = 盤面高さ × 0.5）
- **FR-004**: 持ち駒が表示可能領域に収まる場合、スクロールバーは非表示にしなければならない（もしくは自動制御されること）
- **FR-005**: 先手と後手それぞれの持ち駒が明確に区別して表示されなければならない
- **FR-006**: 持ち駒エリアのスクロールは将棋盤のレイアウトに影響を与えてはならない

### Key Entities

- **持ち駒エリア（先手用 / 後手用）**: 各プレイヤーが取得した駒の一覧を表示する領域。駒の種類と枚数を含む
- **駒リスト行**: 持ち駒エリア内に縦に並ぶ各行。1行 = 1駒種で、駒アイコンと枚数（例：歩 ×4）で構成される

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 対局画面を開いた際、先手の持ち駒エリアが将棋盤の右側・後手の持ち駒エリアが左側に配置されていることを目視で即座（1秒以内）に確認できる
- **SC-002**: 持ち駒エリア内の全ての駒が縦方向に整列して表示されており、駒の種類と枚数を5秒以内に把握できる
- **SC-003**: 持ち駒エリアが将棋盤の高さの半分を超えた場合にスクロール操作で全ての持ち駒を確認でき、エリアの最大高さは盤面高さの半分に制限される
- **SC-004**: 持ち駒の枚数が0〜18枚（理論上の最大）のいずれの場合でも、レイアウトの崩れが発生しない

## Clarifications

### Session 2026-05-22

- Q: 持ち駒の縦並び表示単位（1行あたり何を表示するか） → A: 駒種ごとに1行（駒アイコン + 枚数表示、例：歩 ×4）
- Q: 先手・後手の持ち駒エリアの配置位置 → A: 先手（ユーザー）の持ち駒は将棋盤の右側、後手の持ち駒は将棋盤の左側にそれぞれ配置する
- Q: スクロール発動の基準となる持ち駒エリアの最大高さ → A: 将棋盤の高さの半分を上限とし、超えた場合にエリア内スクロールを有効にする

## Assumptions

- 対象プレイヤーは将棋のルールを知っており、持ち駒エリアの意味を理解している
- 持ち駒エリアのスクロールはマウスホイールまたはタッチスクロールで操作する（スクロールバーのUI形式は実装者に委ねる）
- 先手（ユーザー）の持ち駒エリアは将棋盤の右側、後手の持ち駒エリアは将棋盤の左側にそれぞれ独立して配置する（ユーザー目線での左右対称）
- 現行の将棋盤コンポーネントや持ち駒表示コンポーネントはすでに存在しており、レイアウト変更が主な作業となる
- モバイル対応（縦画面レイアウトへの切り替えなど）は本機能のスコープ外とする
