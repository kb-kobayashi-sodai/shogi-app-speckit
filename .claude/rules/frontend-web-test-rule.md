# テストルール（Vitest / @testing-library/react）

## 目的

ユーザー視点での品質保証を自動化し、リグレッションを防止する。

## ルール一覧

### テストの基本方針

- ユーザーの操作・振る舞いを軸にテストを書く（実装の詳細（内部状態・メソッド名）に依存しない）
- テストはドキュメントとして機能するよう、意図が明確な説明文を書く
- テストファイルは対象ファイルと同じディレクトリに配置する（例: `Board.test.tsx` は `src/components/Board/` と同階層）

### テストの構造

- `describe` でテスト対象のコンポーネント・関数名をグループ化する
- `it` または `test` の説明は「〜の場合、〜になる」の形式で書く（例: `'先手番で合法手マスをクリックすると駒が移動する'`）
- Arrange / Act / Assert（準備・実行・検証）の3フェーズを意識して書く
- 1つの `it` ブロックで1つの振る舞いだけをテストする

### @testing-library/react の使い方

- 要素の取得は `getByRole`, `getByLabelText`, `getByText` を優先する（`getByTestId` は最終手段）
- ユーザー操作には `userEvent` を使用する（`fireEvent` より現実に近い動作）
- 非同期処理のテストは `waitFor` または `findBy*` クエリを使う

### ゲームロジックのテスト

- `src/game/` 以下の純粋関数はユニットテストで網羅する
- 盤面の状態遷移（駒の移動・王手判定・詰み判定）は関数レベルでテストする
- エッジケース（千日手・成り選択・打ち歩詰め・駒打ち）を必ずテストする

### モック

- 外部依存（タイマー）はモックしてよいが、ゲームロジック関数はモックしない
- `vi.useFakeTimers()` はタイマー関連のテストでのみ使用し、テスト後に `vi.useRealTimers()` で戻す

### このプロジェクト固有のテストパターン（参考）

- `src/game/gameEngine.ts` の `processMove`, `createInitialGameState` 等は純粋関数のため、入力状態→出力状態の形でユニットテストする
- `src/game/legalMoves.ts` の合法手計算は、特定の盤面配置を入力として合法手リストを検証する
- `useGame` フックのテストは `renderHook` を使い、アクションディスパッチ後の状態変化を検証する
- `useComputerPlayer` のテストは `vi.useFakeTimers()` を使いタイマーをコントロールする

### 禁止事項

- スナップショットテストのみに頼らない（振る舞いのテストを優先する）
- `console.error` や `console.warn` をテスト内でsilenceしない（警告は修正する）
- テスト間で状態が共有されないよう、`beforeEach` でリセット処理を行う
