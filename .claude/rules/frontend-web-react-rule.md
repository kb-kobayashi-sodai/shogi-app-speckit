# Reactコーディングルール

### コンポーネント設計

- 関数コンポーネントのみ使用する — クラスコンポーネントは禁止
- 1コンポーネント = 1責務。複数の関心事を持つ場合は分割する
- コンポーネントのProps型は必ず定義する（`interface Props` または `type Props`）
- Propsにはデフォルト値を設定できる場合は設定する
- 純粋な表示コンポーネント（状態を持たない）と状態管理コンポーネントを分離する

### Hooks

- カスタムフックは `use` プレフィックスで始める（例: `useGame`, `useComputerPlayer`）
- `useEffect` はできる限り使用しない — イベントハンドラ・レンダー中の計算・ユーザー操作への応答など、`useEffect` なしで表現できる場合はそちらを優先する
- やむを得ず `useEffect` を使う場合は依存配列を必ず正確に記述する（lintの警告を無視しない）
- 副作用のクリーンアップ（タイマー解除、イベントリスナー削除など）を忘れない
- ゲームロジックはカスタムフックに抽出し、コンポーネントからロジックを分離する
- `useState` で管理する状態は最小限にとどめ、派生値はレンダー中の計算で求める

### パフォーマンス

- `useCallback`・`useMemo` などのメモ化はできる限り行わない — 可読性を損なう割に効果が限定的なことが多く、Reactコンパイラが自動最適化できる範囲も広がっている
- コンポーネントを不必要に細分化しない（過度な最適化は可読性を損なう）

### イベントハンドラ

- イベントハンドラ名は `on` + 動詞の形にする（例: `onClick`, `onCellClick`, `onResign`）
- Propsで渡すコールバックは `on` プレフィックス（例: `onModeSelect`, `onPromotionChoice`）

### 禁止事項

- `dangerouslySetInnerHTML` の使用禁止（XSSリスク）
- コンポーネント内でのインライン関数定義を避ける（パフォーマンス劣化を引き起こす場合）
- Propsのバケツリレー（3階層以上）が発生した場合はContextまたはカスタムフックを検討する
